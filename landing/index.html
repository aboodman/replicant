Replicant

Conflict-Free Sync for Mobile Apps

Replicant is a new approach to mobile sync which finally makes it easy — pleasant, even — to offline-enable your mobile application.

Who cares about offline? You should. Because it makes your app instantaneously responsive and reliable under all network conditions, including no network. It permanently banishes spinners and hangs from your user experience. And every interaction with your app feels local, with changes synchronized in the background when there is connectivity.

Features

  • Local-first. All reads and writes go to a local on-device database by default, which is synchronized with the server lazily.
  • Conflict-free. There is zero manual conflict resolution, by either developers or users, for virtually all cases.
  • Convergent. Every client is guaranteed to converge with the server state. It is impossible for two devices to settle into different states.
  • Transactional. Arbitrarily complex transactions are supported and are guaranteed consistent across all nodes.
  • Easy. The data model is a document store, like Firestore, RethinkDB, Mongo, FaunaDB, etc. Transactions are executed serially, like in a single-node transactional database. There is nothing new to learn. You use Replicant like you use any other local datastore, except it also syncs with the server and all other clients.
  • Compatible. Replicant doesn’t demand that you move to a whole new server-side database or stack. It works alongside just about any existing technology.

Technology

Replicant builds on the latest academic research to transform a disconnected set of single-node transactional databases into a Causal-Consistent distributed database. This has been proven to be the highest consistency level possible in a disconnected system. But until now, there have been no commercial systems that implement it for offline synchronization.

While Replicant is technically a CRDT, it differs from currently popular CRDTs in that it: (a) permits arbitrarily complex operations over a rich data model, and (b) guarantees transactional consistency.

What this means is that developers don’t need to adapt their problem to fit a constrained data model. They can use natural data models and simple sequential algorithms, without giving up the key promise of CRDTs: guaranteed convergence.

There is a full design document available to early adopters, which is scheduled for review this year by Jepsen, LLC.
